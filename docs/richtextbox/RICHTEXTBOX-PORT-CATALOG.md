# RichTextBox Port Catalog

This catalog tracks the WPF `RichTextBox` / `System.Windows.Documents`
source-first port in `LeXtudio.Windows`.

`PLAN.md` remains the narrative migration-pattern playbook. `docs/richtextbox/`
(an `index.md` plus one `session*.md` per work session, 42 sessions as of this
refresh) is the detailed session-by-session log. This file is the current
status catalog: what source is linked, what is a local shim, and what is
deliberately deferred.

**Refresh note**: this catalog was badly stale before this pass — it described
a pre-source-port era (local-shell approximations of `RichTextBox`,
`TextEditor`, `TextRange`, etc., referencing files like
`EarlyBatchEditorShims.cs` as the *primary* implementation and an external
`UnoRichText/docs/RichTextBox/session3.md` harness). None of that reflects the
current reality: real upstream WPF source is linked for essentially the entire
editable spine, and `docs/richtextbox/` (42 sessions) is now the live status
log. This refresh was generated by diffing `LeXtudio.Windows.csproj`'s actual
`Compile Include`/`Compile Remove` entries against disk, not by re-reading old
notes.

## Status Key

- `linked-upstream`: upstream WPF source (`ext/wpf` submodule) compiles
  directly, `#if HAS_UNO`-guarded where platform differences require it.
- `local-shim` / `.uno.cs` partial: local file provides Uno-specific behavior
  a linked upstream file depends on (rendering, input forwarding, DP
  registration adjustments, etc.), not a WPF-surface approximation.
- `deferred`: upstream family intentionally excluded from the build via a
  `Compile Remove` wildcard in the `.csproj` — an M5-equivalent scope cut, not
  a bug.

## Upstream Documents/ linkage, by the numbers

Of the **223** `System.Windows.Documents` files in the linked `ext/wpf`
submodule, **146 are actually compiled** (the rest are excluded via `Compile
Remove` wildcards — see "Deferred families" below). This is not a curated
subset of "spine" files with everything else missing — it's essentially the
whole non-deferred WPF document/editing model: `TextContainer`, `TextPointer`,
`TextRange`/`TextRangeBase`, all of `TextRangeEdit*`/`TextEditor*`
(typing, characters, paragraphs, lists, tables, selection, copy/paste,
drag/drop, context menu), the full `Inline`/`Block` family (`Run`, `Span`,
`Paragraph`, `List`, `ListItem`, `Table*`, `Hyperlink`, `Bold`/`Italic`/
`Underline`), `EditingCommands`, undo (`TextTreeUndo*`, `ParentUndoUnit`,
`ChangeBlockUndoRecord`), and `TextRangeSerialization`.

`AnchoredBlock.cs`, `FlowDocument.cs`, `TextElementCollection.cs`,
`TextRange.cs`, `TextRangeEdit.cs`, `TextRangeEditLists.cs`, `TextSchema.cs`
are dead code on disk, kept only because deleting unused files isn't done
speculatively). Notable active ones and what they actually do (not
approximations of WPF surface — narrow, targeted bridges):

| `DeferredRunTextReference.cs` | `Run.Text`'s lazy-value resolver; matches upstream's `TextRangeBase.GetTextInternal` approach exactly since session 33 (previously miscoded to go through full `TextRange` construction, which crashed on `TextRangeEditTables`' ancestor walk). |

Outside `System.Windows/Documents/` but load-bearing for the same spine:
`System.Windows/LogicalTreeHelper.cs` (fixed in session 35 — previously a

walking `Parent` chains, including `TextRangeEditTables`/`TextRangeEditLists`),
`System.Windows/Controls/RichTextBox.uno.cs` (pointer/keyboard input

`MS.Internal.Documents/FlowDocumentView.uno.cs` + `UnoFlowDocumentTextView.cs`
(the actual rendering/hit-testing host — `FlorenceLayoutEngine`-based line/run

## Deferred families (excluded via `Compile Remove`)

These upstream `Documents/` families are excluded from the build entirely —
this mirrors WPF-DataGrid's M5-style "deferred unless a consumer requires it"
scope, not a bug:

- **Fixed/paginated documents**: `Fixed*.cs` (54 files — `FixedDocument`,
  `FixedPage`, the `FixedSOM*` semantic-analysis family, `FixedFindEngine`,
  etc.), `DocumentSequence*.cs`, `PageContent*.cs`, `ChildDocumentBlock.cs`,
  `DocumentReference*.cs`, `WpfPayload.cs`, `XPSS0ValidatingLoader.cs`.
- **Speller / TSF / IME**: `Speller*.cs`, `TextServices*.cs`,
  `WinRTSpellerInterop*.cs`, `NaturalLanguageHyphenator.cs`,
  `NLGSpellerInterop.cs`, `ImmComposition.cs`, `TextStore.cs`.
- **Win32-specific adorner/event plumbing**: `Adorner*.cs`, `CaretElement.cs`
  (Uno's own `FlowDocumentView` caret replaces this), `ColumnResize*.cs`,
  `CompositionAdorner.cs`, `MoveSizeWinEventHandler.cs`, `WinEventHandler.cs`.
- **Misc niche/legacy internals**: `Glyphs.cs`, `RubberbandSelector.cs`,
  `HighlightVisual.cs`, `TextFlow.cs`, `FlowNode.cs`, `FlowPosition.cs`.

Also intentionally unsupported (not excluded from compilation, but gated
`#if HAS_UNO` inside linked files rather than via `Compile Remove`):
`TextRangeBase.CanSave`/`CanLoad`/`Save`/`Load` only recognize
`DataFormats.Text` — `Xaml`/`Rtf`/`XamlPackage` throw a predictable
`ArgumentException` (`docs/richtextbox/session34.md`), and
`List.Apply` (new-list creation from plain paragraphs via
`TextRangeEditLists.ConvertParagraphsToListItems`) throws `NotSupportedException`
(`docs/richtextbox/session37.md`) — editing an *existing* list (construction,
indent/outdent, marker removal, bullet/number toggling) works correctly and is
tested (`docs/richtextbox/session38.md`-`session40.md`).

## IME integration (session 43)

RichTextBox now integrates real OS-level IME composition via
`LeXtudio.UI.Text.Core` (NuGet, `src/Directory.Packages.props`), a
UI-framework-agnostic cross-platform bridge (`CoreTextEditContext` +
native macOS-AppKit/Linux-IBus/Win32 adapters) also used by the sibling
UnoEdit project — **not** a revival of WPF's own excluded
TSF/TextServices/ImmComposition family. See
`src/LeXtudio.Windows/System.Windows/Controls/RichTextBox.Ime.uno.cs` and
`docs/richtextbox/session43.md` for the integration shape, the offset-space
bug found and fixed (IME range offsets are plain-text-string indices, not
raw `TextContainer` symbol offsets), and the verification methodology
(`CoreTextEditContext`'s own public `Raise*` methods invoked directly in
tests, since real OS keystroke-driven composition can't be scripted
headlessly). Offset mapping (`GetPlainTextOffset`/`GetPositionAtPlainTextOffset`
in `RichTextBox.Ime.uno.cs`) works generally for any document structure the
`.Text` getter already handles correctly (paragraphs, lists, tables) — it
reuses that getter's forward mapping and inverts it via binary search rather
than reimplementing WPF's plain-text walk by hand, and is verified for
multi-paragraph documents.

## Open threads

- List indentation/marker-removal/toggle algorithms all work correctly on an
  *existing* list (sessions 37-40) now that session 35's `LogicalTreeHelper`
  fix is in place; only new-list *creation* (`List.Apply`) remains
  unsupported.
- Table construction (session 43) does not crash and the document/editing
  model reads back correctly, but tables have **no visual rendering at all**
  — `FlorenceLayoutEngine.Format` only walks `Paragraph` blocks, so `List`
  and `Table` content is invisible on screen regardless of any `Parent`-chain
  fix (this also means sessions 37-40's List tests only ever validated the
  document/editing model, not visual rendering). `TableCell`/`TableRow`/
  `TableRowGroup`'s `OnNewParent` override (real WPF's hook for syncing a
  reparented cell into its row's `Cells` collection) is never invoked by this
  shim's `LogicalTreeHelper`, unlike `ListItem` which doesn't need it — a
  narrower latent risk than session 35's original bug, since
  `TableCellCollection.Add`/etc. maintain their own storage directly and
  don't depend on the `OnNewParent` side effect for the normal construction/
  editing path. Not chased further: no crash found, no rendering to verify
  against, and no current consumer need.
- IME composition offset mapping now works generally (see "IME integration"
  above); `CompositionStarted`/`CompositionCompleted` still have no visual
  underline/highlight for in-progress composition text (cosmetic).
- `docs/richtextbox/index.md`'s milestone-driven backlog is largely exhausted
  as of session 42 — next work should be prioritized against actual consumer
  needs rather than continuing to mine the milestone list speculatively.
